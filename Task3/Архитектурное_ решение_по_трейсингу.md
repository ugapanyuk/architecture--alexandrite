# Задание 3. Трейсинг

## 1. Анализ системы и точек отказа

Текущая архитектура представляет собой распределенную систему, состоящую из нескольких независимых сервисов, которые общаются как синхронно (по HTTP), так и асинхронно (через RabbitMQ). Именно на стыках этих взаимодействий возникают «слепые зоны», где запрос или бизнес-процесс (в нашем случае — заказ) может замедлиться, завершиться с ошибкой или полностью потеряться.

**Ключевые точки отказа и потери контекста:**

1.  **Синхронные вызовы между сервисами:**
    *   `Internet Shop` (Frontend) → `Shop API` (Backend)
    *   `CRM` (Frontend) → `CRM API` (Backend)
    *   `MES` (Frontend) → `MES API` (Backend)
    *   `Shop API` → `3d files storage` (S3)
    *   `MES API` → `3d files storage` (S3)

2.  **Асинхронное взаимодействие (самая критичная зона):**
    *   `CRM API` → `Messages Queue` (RabbitMQ): Публикация сообщения о новом заказе.
    *   `Messages Queue` → `MES API`: Потребление сообщения для расчета стоимости.
    *   `MES API` → `Messages Queue`: Публикация сообщения с рассчитанной стоимостью.
    *   `Messages Queue` → `CRM API`: Потребление сообщения для создания B2B-заказа.

3.  **Взаимодействие с базами данных:**
    *   `Shop API` → `Shop DB`
    *   `MES API` → `MES DB`

Чтобы получить полную картину прохождения заказа, необходимо **покрыть трейсингом все бэкенд-сервисы**: `Shop API`, `CRM API` и `MES API`, а также обеспечить проброс контекста трассировки через `Messages Queue`.

## 2. Мотивация

Внедрение распределенной трассировки напрямую решает главную проблему бизнеса — **потерю заказов и отсутствие понимания, на каком этапе они находятся**. Вместо того чтобы часами анализировать разрозненные логи нескольких систем, команда поддержки или разработки сможет одним кликом увидеть всю историю прохождения заказа.

**Влияние на технические и бизнес-метрики:**

1.  **Mean Time To Resolution (MTTR) / Среднее время решения инцидентов (Техническая):** Значительно сократится. Инженеры будут сразу видеть проблемный сервис и контекст ошибки. 
2.  **Order Fulfillment Time / Полное время выполнения заказа (Бизнес):** Мы сможем точно измерять, сколько времени заказ проводит на каждом этапе (`PRICE_CALCULATED`, `MANUFACTURING_APPROVED` и т.д.). Это позволит выявить узкие места и оптимизировать их.
3.  **Error Rate Visibility / Видимость частоты ошибок (Техническая):** Трейсинг покажет не только ошибки 500, но и логические ошибки внутри бизнес-процесса (например, заказ не был создан после расчета цены).
4.  **Customer Satisfaction (CSAT) / Удовлетворенность клиентов (Бизнес):** Быстрое решение проблем и сокращение количества «потерянных» заказов напрямую повысит лояльность как B2C, так и B2B клиентов.
5.  **B2B Partner Churn Rate / Отток B2B-партнеров (Бизнес):** Предоставляя партнерам прозрачность и стабильность API (подкрепленную данными из трейсинга), мы снизим их отток.

## 4. Предлагаемое решение

Мы предлагаем внедрить стек технологий, основанный на открытом стандарте **OpenTelemetry**, который является де-факто стандартом индустрии для сбора телеметрии.

**Компоненты решения:**

1.  **OpenTelemetry SDK:** Клиентские библиотеки, которые будут встроены в каждый бэкенд-сервис (`Shop API` на Java, `CRM API` на Java, `MES API` на C#). Они будут отвечать за создание спанов и их отправку. Важно, что SDK имеют встроенную инструментацию для популярных фреймворков (Spring, ASP.NET Core), баз данных и систем очередей, что упростит внедрение.
2.  **Использование особенностей RabbitMQ:** Ключевая доработка. При отправке сообщения в очередь (`CRM API` → `RabbitMQ`) SDK будет автоматически добавлять в заголовки сообщения идентификатор трейса (`trace_id`). При получении сообщения (`MES API` ← `RabbitMQ`) SDK будет считывать этот заголовок и продолжать существующий трейс, а не начинать новый.
3.  **OpenTelemetry Collector (OTel Collector):** Централизованный агент для сбора, обработки и экспорта телеметрии. Сервисы будут отправлять данные не напрямую в систему хранения, а в коллектор. Это повышает гибкость и надежность.
4.  **Jaeger:** Открытая и популярная система для хранения и визуализации трейсов. Jaeger предоставляет удобный UI для поиска, анализа и визуализации полного пути прохождения запросов.

## 5. Компромиссы

1.  **Производительность:** Модификация кода для сбора трейсов добавляет небольшую, но измеримую нагрузку на приложения. Для минимизации этого эффекта на высоконагруженных системах можно будет внедрить сэмплирование  (сохранение только определенного процента трейсов, например, 10% всех запросов и 100% запросов с ошибками).
2.  **Стоимость хранения:** Данные трейсинга занимают значительный объем дискового пространства. Необходимо будет настроить политику хранения данных (retention policy) в Jaeger, например, хранить полные трейсы не более 15 дней.
3.  **Сложность внедрения:** Хотя OpenTelemetry упрощает процесс, потребуется время разработчиков на интеграцию SDK во все сервисы, особенно на корректную настройку проброса контекста через RabbitMQ.
4.  **Трейсинг внешних систем:** Мы не можем трассировать запросы внутри управляемых сервисов, таких как Yandex Cloud S3 или PostgreSQL. Мы можем видеть только спан, который отражает вызов к этому сервису из нашего приложения, его длительность и результат. Этого, однако, достаточно для большинства сценариев отладки.

## 6. Аспекты безопасности

1.  **Доступ к UI:** Доступ к веб-интерфейсу Jaeger будет ограничен внутренней сетью компании. Для входа будет настроена аутентификация через корпоративный SSO (Single Sign-On) или, как минимум, через reverse-proxy с базовой аутентификацией. Доступ будет предоставляться только инженерной команде и команде поддержки.
2.  **Сетевая изоляция:** Все компоненты системы трейсинга (OTel Collector, Jaeger) будут развернуты в приватной подсети в Yandex Cloud, недоступной из публичного интернета. Приложения будут отправлять данные в коллектор по внутренней сети.
3.  **Маскирование чувствительных данных:** Трейсы могут содержать персональные данные (например, в параметрах HTTP-запросов или ответах). В OTel Collector будет настроен процессор `attributes`, который будет находить и маскировать (например, заменять на `[REDACTED]`) чувствительную информацию по заранее определенным правилам и ключам, чтобы избежать ее сохранения в системе трейсинга.

### 7. Дополнительное задание: Предлагаемое решение

Простого просмотра трейсов недостаточно для проактивной работы. Мы можем использовать данные из трейсинга для автоматического обнаружения "зависших" заказов.

**Реализация:**

1.  **Trace Analysis Service (Сервис анализа трейсов):** Это будет новый микросервис. Он будет получать копию всех трейсов (например, из OTel Collector или напрямую из Jaeger Storage).
2.  **Логика сервиса:** Сервис анализирует завершенные трейсы. Он ищет аномалии, например:
    *   Трейс с тегом `order.id` начался, но не дошел до определенного этапа (например, нет спана `MANUFACTURING_STARTED`) в течение N часов.
    *   Трейс содержит спан с ошибкой (`error=true`) на критическом этапе (например, при подтверждении заказа в CRM).
3.  **Интеграция с мониторингом:** При обнаружении аномалии `Trace Analysis Service` инкрементирует специальную метрику в Prometheus (например, `alexandrite_stuck_orders_total{reason="timeout"}`).
4.  **Алертинг:** Настроенный ранее `Alertmanager` будет отслеживать эту метрику. Если ее значение становится больше 0, он отправляет критический алерт команде поддержки с указанием `order.id`, который нужно проверить.
