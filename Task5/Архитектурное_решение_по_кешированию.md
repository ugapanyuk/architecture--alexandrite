# Задание 5. Кеширование

## 1. Мотивация

Основная проблема, с которой мы сталкиваемся в MES, — это критическая деградация производительности дашборда операторов. Страница со списком заказов загружается очень медленно, что напрямую влияет на продуктивность и вознаграждение сотрудников. Причина кроется в том, что для отображения этого списка `MES API` выполняет тяжелые, повторяющиеся запросы к базе данных (`MES DB`) при каждом обновлении страницы или смене фильтра. Нагрузка на БД постоянно растет, что замедляет всю систему.

Внедрение кеширования позволит решить эту проблему, значительно снизив нагрузку на базу данных. Мы будем хранить часто запрашиваемые данные (списки заказов) в быстрой in-memory базе данных (кеше). Это приведет к следующим улучшениям:

*   **Ускорение ответа API:** `MES API` будет быстро отдавать данные из кеша, вместо того чтобы ждать ответа от базы данных. Это напрямую ускорит загрузку страницы для операторов.
*   **Снижение нагрузки на базу данных:** Количество read-запросов к `MES DB` сократится на порядок, что высвободит ее ресурсы для действительно важных операций (запись, обновление статусов) и повысит общую стабильность системы.
*   **Повышение масштабируемости:** Система сможет выдерживать большее количество операторов, одновременно работающих с дашбордом, без деградации производительности.

**Элементы для кеширования:**
Основной и единственный кандидат на кеширование в данном контексте — это результаты запросов `MES API` на получение списков заказов по статусам, которые отображаются на дашборде операторов.

## 2. Предлагаемое решение

### Выбор типа кеширования

Предлагается внедрять серверное кеширование.

**Клиентское кеширование** (в браузере) в данном случае не подходит. Основная задача операторов — видеть актуальный список доступных заказов и быстро брать их в работу. Если один оператор возьмет заказ, остальные должны немедленно увидеть это изменение. Клиентский кеш не может обеспечить такую согласованность данных между разными пользователями, что приведет к конфликтам и путанице.

**Серверный кеш** (например, с использованием Redis) будет единым для всех экземпляров `MES API`, обеспечивая централизованное и согласованное хранилище данных.

### Паттерн кеширования

Будем использовать паттерн **Cache-Aside (Lazy Loading)**.

**Как это работает:**
1.  Приложение (`MES API`) получает запрос на список заказов.
2.  Оно обращается к кешу.
3.  **Cache Hit (Попадание):** Если данные есть в кеше, приложение немедленно возвращает их пользователю.
4.  **Cache Miss (Промах):** Если данных в кеше нет, приложение выполняет запрос к базе данных, получает данные, сохраняет их в кеш (с установленным временем жизни) и только после этого возвращает пользователю.

**Почему Cache-Aside, а не другие паттерны?**

*   **Простота и надежность:** Это самый простой в реализации и самый отказоустойчивый паттерн. Если кеш по какой-то причине недоступен, система продолжит работать, просто медленнее (данные будут браться напрямую из БД). Для нас это идеальный стартовый вариант.
*   **Write-Through / Write-Back:** Эти паттерны усложняют логику записи и добавляют задержку (latency) в операции обновления статуса заказа, что нам не нужно. Наша проблема — медленное чтение, а не запись.
*   **Refresh-Ahead:** Этот паттерн проактивно обновляет данные в кеше до их истечения. Он сложнее в реализации и избыточен для нашей задачи. Нам не нужно предсказывать, какой список понадобится оператору; достаточно быстро загрузить его по запросу.

### Стратегия инвалидации кеша

Неактуальные данные в кеше — главная проблема, которую нужно решить. Можно использовать гибридную стратегию, комбинирующую программную инвалидацию и короткое время жизни ключа (TTL).

| Стратегия | Описание | Плюсы | Минусы | Применимость для нас |
| :--- | :--- | :--- | :--- | :--- |
| **Программная инвалидация + TTL (Выбранный вариант)** | При любом изменении данных (обновление статуса заказа) приложение явно удаляет из кеша ключи, которые стали неактуальными. TTL (например, 30 секунд) используется как защитный механизм на случай, если инвалидация не сработала. | **Максимальная актуальность данных.** Операторы почти мгновенно видят изменения. TTL гарантирует, что кеш в любом случае очистится через короткое время. | Сложнее в реализации, так как требует дополнительной логики в коде, который изменяет данные. | **Идеально подходит.** Обеспечивает скорость и актуальность, что критично для работы операторов. |
| **Только TTL** | Каждая запись в кеше живет строго определенное время (например, 1 минута), после чего автоматически удаляется. | **Очень просто в реализации.** Не требует никакой дополнительной логики. | Данные могут быть неактуальными в течение всего времени жизни ключа. Оператор может не видеть новый заказ целую минуту. | **Не подходит.** Слишком большие задержки в обновлении данных для нашей бизнес-задачи. |
| **Инвалидация по ключу (без TTL)**| Только программное удаление ключей при изменении данных. Время жизни не устанавливается. | Данные всегда актуальны, если логика инвалидации работает без сбоев. | **Риск "вечного" кеша.** Если по какой-то причине логика инвалидации не сработает (баг, сбой), неактуальные данные останутся в кеше навсегда (до перезапуска). | **Слишком рискованно.** Отсутствие защитного механизма в виде TTL может привести к серьезным проблемам. |

**Выбранная стратегия в деталях:**
1.  **Ключ кеширования:** Ключ будет формироваться на основе параметров запроса. Например, `orders:status:MANUFACTURING_APPROVED:page:1`.
2.  **Процесс записи/обновления:** Когда оператор берет заказ в работу, его статус меняется с `MANUFACTURING_APPROVED` на `MANUFACTURING_STARTED`. В этот момент `MES API` после успешного обновления записи в `MES DB` должен принудительно удалить (инвалидировать) как минимум два ключа из кеша:
    *   `orders:status:MANUFACTURING_APPROVED:*` (все страницы этого списка)
    *   `orders:status:MANUFACTURING_STARTED:*` (все страницы этого списка)
3.  **TTL (Time-To-Live):** На все ключи в кеше будет установлено короткое время жизни, например, **30 секунд**. Это гарантирует, что даже если программная инвалидация по какой-то причине не сработает, пользователи увидят обновленные данные с максимальной задержкой в 30 секунд.
